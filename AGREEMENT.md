# slide-upa 合意一覧（改訂版）

**更新:** 2026-06-08  
**前提:** 台本は既存 `script-upa`（会話形式）とは**別物**。佐藤さんが**一人で話す**用の台本からスライドを生成する。ウパ博士・パニっくんは**台本の会話としては使わない**が、**スライドの見た目（世界観）** では使う。

---

## 1. 目的・スコープ

| 項目 | 内容 |
| --- | --- |
| **主用途** | MTG 中の画面共有（録画の背景資料） |
| **今回やること** | 一人語り台本 → ウパ博士世界観の HTML スライドを生成する **slide-upa** |
| **やらないこと** | ポータル改修、台本作成スキル本体、振り返り図解ツール、既存 script-upa / diagram-upa の改修 |
| **ポータル** | 載せる必要があれば URL を手動共有。slide-upa v1 の範囲外 |

---

## 2. 教育フローでの位置

```
① 台本作成スキル（新規・一人語り） → script.md 等
② slide-upa（本ドキュメント）         → スライド（index + presenter）
③ 振り返り図解ツール（別途）         → 図解（会話形式でも可）
④ ポータル反映（必要時・手動）
```

- **script-upa（既存）** … 今回のフローでは使わない
- **diagram-upa（既存）** … 振り返り図解用。入力・ツールとも slide-upa とは別

## 3. プロジェクト構成

| 項目 | 内容 |
| --- | --- |
| **名前** | `slide-upa` |
| **場所** | `src/slide-upa/`（`upahakase-visual-explainers` の配下ではない） |
| **形** | Cursor スキル（`.claude/skills/slide-upa/`） |
| **キャラ画像** | `slide-upa/assets/characters/` に PNG を置く（**単体完結**）。初回は `upahakase-visual-explainers` から**コピーして**用意する。**図解リポ側の画像ファイルは読み取り・コピー元としてのみ使い、編集・移動・削除しない** |
| **成果物** | `output/{プロジェクト名}/index.html`（共有用）＋ `presenter.html`（自分用）＋ `images/` |
| **公開 URL** | `slides-{プロジェクト名}-{YYYY-MM-DD}.surge.sh` |
| **デザイン正本** | `slide-upa` 内のスライド専用完成見本 HTML。色・フォントは diagram-upa を**参照**（コピーしてスライド用に改造） |

## 4. 入力（一人語り台本）

### 4.1 指定の仕方

- ユーザーが **ファイルパスを指定**（例：「この `script.md` からスライド作って」）
- 場所は固定しない（どこにあってもよい）

### 4.2 見出し構造（必須）

```markdown
# タイトル
## 第1章：章タイトル
### 1-1. 節見出し

（節の中身＝一人語り原稿）
```

| 見出し | 役割 |
| --- | --- |
| `#` | タイトル → **表紙スライド** |
| `##` | 章 |
| `###` | 節 → **基本はここでスライド切替**（v1 は ### 単位で試す） |

### 4.3 節の中身

- **普通の段落** と **箇条書き** の**両方**使ってよい（C）
- **会話形式にしない**（`**パニっくん：**` / `**ウパ博士：**` は使わない）
- script-upa 専用の `## パニっくんとウパ博士の対話` 行も**不要**

### 4.4 形式チェック

- 上記ルールに合わない場合 → **スライドを生成せず、直し方を表示**して止める
- 事実確認（Web 検証）は**しない**（台本が正）

---

## 5. 台本 → スライドのルール

### 5.1 切替・枚数

| ルール | 内容 |
| --- | --- |
| 基本 | `###` 見出し **1つ = スライド 1枚** |
| 表紙 | 台本先頭の `# タイトル` から **1枚**（ウパ博士の絵） |
| 例外 | `###` が1つもない章は `##` 見出しで **1枚** |
| 枚数上限 | **なし**（台本の見出しとマーカーで決まる） |
| 自動ビジュアル判定 | **しない**（`[visual]` は手動指定のみ） |

### 5.2 レイアウト型（マーカー優先）

v1 の型は **4種類**。足りなければ試運用後に追加。

| 型 | いつ使うか |
| --- | --- |
| ① 表紙 | `# タイトル` |
| ② 要点リスト | マーカーなしの `###`（節内原稿を **自動要約**） |
| ③ 大きな一言 | `[quote: セリフ]` がある `###` |
| ④ 画像メイン | `[visual]` がある `###` |

---

## 6. マーカー

### 6.1 一覧

| マーカー | 書き方 | 意味 |
| --- | --- | --- |
| 画像多め | 見出し直後の行に `[visual]` | この節は④画像メイン型 |
| 画像内容 | `[image: 内容の説明]` | AI/キャラ画像の指示（`[visual]` の下） |
| 強調一言 | `[quote: 話す一言]` | ③大きな一言型（**話し手が強調するフレーズ**） |

### 6.2 例

```markdown
### 2-1. 自動報告の仕組み

今日お伝えするのは、毎朝 Slack に進捗が届く仕組みです。

- 前日のタスクを AI が読む
- 完了・未完了を整理する
- 決まった時間に投稿する

[quote: ポイントは「人が毎回書かなくていい」ことです]
```

```markdown
### 2-2. 全体の流れ
[visual]
[image: 3ステップの矢印図。Slack アイコンあり]
[image: スマホを見て安心するパニっくん]
```

### 6.3 エラー・警告

| 状況 | 動作 |
| --- | --- |
| `[visual]` あり、`[image:]` なし | **警告して生成しない**（修正後に再実行） |
| `[visual]` なし、`[image:]` あり | **無視＋警告**（生成は続行） |
| 同じ `###` 内に `[image:]` 複数行 | **1枚のスライドにまとめる** |
| `[quote:]` なし | ③は作らない（②要点リストのみ） |

## 7. デザイン・キャラクター

| 項目 | 内容 |
| --- | --- |
| キャラ PNG の正本 | `slide-upa/assets/characters/`（slide-upa 内で完結。シンボリックリンクは使わない） |
| 初回セットアップ | `upahakase-visual-explainers` 側の PNG を**コピー**して `assets/characters/` に置く |
| 図解リポ側 | **変更しない**（コピー元として参照するだけ） |
| デプロイ時 | `assets/characters/` から `output/{名}/images/` へ必要ファイルを**実ファイルとしてコピー**して surge に載せる |
| 表紙 | ウパ博士の絵で世界観を出す（`assets/characters/` の PNG を使用） |
| 各枚 | **右下**にページ番号（例：`3/18`）、**左下**に小さなウパ博士アイコン |
| 枠線 | スライド上下にウパ博士**ブランドカラー**の線 |
| 著作権フッター | スライドには**載せない**（契約書でカバー） |
| まとめスライド | 特別指定なし。台本構成に従う |
| AI 画像 | `[image:]` の指示に従う。キャラ優先だが**細かいルールは試運用後に調整** |
| アスペクト比 | 16:9（画面共有向け） |

## 8. MTG 当日の使い方（2ファイル + 同期）

| ファイル | 誰が見る | 内容 |
| --- | --- | --- |
| `index.html` | クライアント（画面共有） | スライドのみ |
| `presenter.html` | 佐藤さんだけ | スライド ＋ **その枚の一人語り原稿** |

### 手順

1. **`presenter.html` を先に開く**
2. 「**共有用を開く**」ボタン → `index.html` が別タブで開き、**同期開始**
3. Google Meet では **`index.html` のタブだけ**画面共有
4. **枚送り（次へ/前へ）** は presenter 側で操作 → 共有タブも追従
5. MTG 中のスライド文言編集は**しない**（v1）

### ローカル確認（デプロイ前）

- `npx serve output/{名}` 等で **http://** 経由でブラウザ確認（2タブ同期のため file:// は避ける）

## 9. 生成フロー

```
台本パス指定 → 形式チェック → ワンショット生成
  → ローカルでブラウザ確認（presenter + index）
  → OK → surge デプロイ → URL を返す
```

- ユーザーへの報告は **URL と短い要約** が中心
- ポータル YAML の書き方は v1 スキル内に**手順メモ**として書いてもよい（コード改修は別）
